# Session: Phân tích & Cải thiện Chất lượng Markdown + UX Xử lý
Date: 2026-05-18
Status: Active

## Background
Người dùng phát hiện **2 vấn đề nghiêm trọng** trong hệ thống DocuMark AI Editor:
1. **Chất lượng Markdown kém** — Kết quả chuyển đổi từ PDF sang Markdown bị sai cấu trúc bảng, mất bố cục, cột bị trộn lẫn.
2. **UX quá trình xử lý rất tệ** — Giao diện chỉ hiển thị "Đang tải lên server..." và treo ở bước 1, không cập nhật tiến trình → người dùng tưởng hệ thống bị treo.

## Topics Discussed

### Topic 1: Chất lượng Markdown Output kém

#### Phân tích gốc rễ (Root Cause Analysis)

**Vấn đề phát hiện từ output mẫu:**
- Bảng "Thông tin chung học phần" (Section A): **RENDER TỐT** — bảng 2 cột đơn giản, Docling xử lý chính xác
- Bảng "Cấu trúc học phần" (Section E): **RENDER XẤU** — bảng phức tạp 13+ cột, nhiều header rows merged → Docling tạo ra bảng quá nhiều cột rỗng, cấu trúc loạn
- Bảng "Kế hoạch & Phương pháp" (Section F): **RENDER CỰC XẤU** — nội dung dài bị gộp vào 1 cell, không xuống dòng, bảng trải dài ngang → không đọc được

**Nguyên nhân kỹ thuật:**
1. **Docling `DocumentConverter()` đang dùng cấu hình mặc định** → `TableFormerMode.FAST` (nhanh nhưng thiếu chính xác)
2. **Không bật `do_cell_matching`** → Các cell bị merged trong PDF gốc không được xử lý đúng
3. **Không bật `do_table_structure`** → Docling có thể miss các bảng phức tạp
4. **Không có post-processing** → Markdown output thô, không được clean up sau khi Docling xuất ra

#### Giải pháp đề xuất

**A. Cấu hình Docling nâng cao (Backend):**
```python
from docling.datamodel.base_models import InputFormat
from docling.datamodel.pipeline_options import PdfPipelineOptions, TableFormerMode
from docling.document_converter import DocumentConverter, PdfFormatOption

pipeline_options = PdfPipelineOptions()
pipeline_options.do_table_structure = True
pipeline_options.table_structure_options.mode = TableFormerMode.ACCURATE
pipeline_options.table_structure_options.do_cell_matching = True

converter = DocumentConverter(
    format_options={
        InputFormat.PDF: PdfFormatOption(pipeline_options=pipeline_options)
    }
)
```

**B. Post-processing Markdown:**
- Loại bỏ cột rỗng thừa trong bảng
- Normalize line breaks bên trong cell
- Cleanup trailing spaces và pipe alignment
- Wrap text dài trong cell để dễ đọc

**C. Export Alternatives:**
- Ngoài `export_to_markdown()`, xem xét tùy chọn export custom cho bảng phức tạp
- Với bảng >8 cột: tự động chuyển sang format text có cấu trúc thay vì bảng markdown

### Topic 2: UX quá trình xử lý — "Treo ở bước 1"

#### Phân tích gốc rễ (Root Cause Analysis)

**Luồng xử lý hiện tại:**
1. Frontend gửi file qua `axios.post()` (không có progress callback)
2. Backend nhận file → `convert_document_to_markdown()` (synchronous, blocking)
3. Toàn bộ thời gian xử lý (~30s-2 phút) nằm trong 1 HTTP request duy nhất
4. Frontend chỉ set stage = `'uploading'` rồi chuyển ngay sang `'extracting'` — **nhưng cả 2 stage xảy ra trong cùng 1 `await`**
5. Kết quả: Stepper UI hiển thị **4 bước** (`Đọc PDF → Kết nối AI → Xử lý → Định dạng`) nhưng thực tế chỉ có **1 action duy nhất** → UI bị đánh lừa, treo ở bước "Kết nối AI" suốt quá trình

**Vấn đề code cụ thể:**
- `ProcessingStatus.tsx` hiển thị 4 steps: `extracting`, `uploading`, `generating`, `formatting`
- `types.ts` chỉ khai báo: `'idle' | 'uploading' | 'extracting' | 'complete'` — **không match** với các step ID trong UI
- `App.tsx` chỉ dispatch 2 lần: `stage: 'uploading'` → `stage: 'extracting'` → rồi `await` cho đến khi xong → `stage: 'complete'`
- **Không có SSE/WebSocket/polling** → không thể biết backend đang ở đâu

#### Giải pháp đề xuất

**A. Server-Sent Events (SSE) cho progress streaming:**
- Backend tạo endpoint SSE `/api/convert-stream`
- Chạy conversion trong background thread
- Push progress events qua SSE: `uploading → reading → table_analysis → formatting → complete`

**B. Frontend progress simulation (giải pháp nhanh):**
- Nếu chưa muốn refactor SSE, dùng **simulated progress** với timer
- Tự động chuyển qua các stages dựa trên thời gian ước tính
- Hiển thị logs giả lập nhưng thực tế: "Đang phân tích cấu trúc trang...", "Đang nhận diện bảng...", v.v.
- Khi `await` resolve → nhảy thẳng sang `complete`

**C. Upload progress (chunk):**
- Sử dụng `axios.onUploadProgress` để hiển thị % upload thật sự
- Giúp người dùng thấy tiến trình thực tế ở bước 1

**D. Cải thiện ProcessingStatus UI:**
- Fix mismatch giữa `types.ts` stages và `ProcessingStatus.tsx` steps
- Thêm estimated time remaining
- Thêm log messages chi tiết hơn
- Thêm nút "Cancel" để hủy xử lý

### Topic 3: Deep-dive — Tại sao bảng Markdown bị render xấu? (trao đổi thêm 2026-05-18 09:10)

Phân tích cụ thể 3 loại bảng trong output PDF đề cương IoT:

#### Bảng A (Section A — "Thông tin chung học phần") → ✅ Render TỐT
- 2 cột đơn giản, Docling xử lý chính xác.
- Tuy nhiên vẫn có lỗi nhỏ: dòng 8 (`Ngành đào tạo  Loại học phần`) và dòng 16 (`Thực hành  Thí nghiệm  Bài tập`) — nhiều dòng riêng biệt bị gộp vào 1 cell bằng double-space.

#### Bảng E (Section E — "Cấu trúc học phần") → ❌ Render XẤU
- **Multi-row header (3 dòng):** `STT | Nội dung | CĐR | Số tiết | Phân bố thời gian` → sub-header `Lên lớp | Phân bố lên lớp` → sub-sub `LT | TL | TH | BT | TN`
- Docling **không hiểu merge** → tạo ra **13 cột flat** thay vì cấu trúc phân cấp
- **8 cột rỗng thừa** ở cuối mỗi dòng dữ liệu
- Dữ liệu rải rác, hầu hết cell là `|   |`

#### Bảng F (Section F — "Kế hoạch & Phương pháp") → ❌❌ Render CỰC XẤU
- Mỗi cell chứa **20-30 dòng text** → Docling xuất thành **1 dòng 800+ ký tự**
- Double-space `  ` làm separator nhưng hầu hết renderers **không** render thành newline bên trong cell
- Bảng kéo dài hàng ngàn pixel theo chiều ngang → **không đọc được**
- Merged cell "Hoạt động dạy và học" (chứa 2 sub-column: Giảng viên + Sinh viên) bị phá vỡ cấu trúc

#### Kết luận: 3 điểm yếu cần fix đồng thời
| # | Điểm yếu | Hậu quả |
|---|-----------|---------|
| 1 | `TableFormerMode.FAST` | Bỏ qua merged cells, multi-row headers |
| 2 | `do_cell_matching = false` | Dữ liệu rơi sai cột, cột rỗng thừa |
| 3 | Không post-processing | Text dài trên 1 dòng, bảng quá rộng |

#### Ý tưởng mới: "Smart Table Detection"
- Với bảng >8 cột hoặc cell >200 ký tự → tự động chuyển sang **structured list** thay vì bảng markdown
- Ví dụ: Bảng F chuyển thành `### Bài 1 (5 tiết)` → danh sách `**Nội dung:** ...`, `**Giảng viên:** ...`, `**Sinh viên:** ...`
- Đọc được **rất rõ ràng** so với 1 dòng 800 ký tự trong bảng

### Topic 4: DOCX Tables — Nội dung biến mất hoàn toàn (2026-05-18)

**Triệu chứng báo cáo:** Convert file `.docx` → markdown bị **mất toàn bộ nội dung tất cả các bảng**. Đặc biệt nghiêm trọng với bảng có:
- Merged cells dọc (ô gộp nhiều dòng, ví dụ: "Bài 2 (3)" span ~15 dòng)
- Nested content trong cell (nhiều đoạn, bullet, sub-sections như "2.1", "2.1.1")
- Bold/mixed formatting trong cell
- Bảng >6 cột

**Screenshot minh họa:** Bảng đề cương môn học — cột "Bài 2 (3)" merge dọc, cột nội dung có "LÝ THUYẾT: 03 tiết" + danh sách 2.1 → 2.2.2, cột phương pháp dạy học dày đặc text bold + plain.

#### Root Cause Analysis — Bug trong `markdown_cleaner.py`

**Luồng xử lý bị lỗi:**

```
Docling convert .docx
     ↓
export_to_markdown() → bảng pipe format bình thường (có content)
     ↓
clean_markdown() → _clean_tables()
     ↓
_is_complex_table() = TRUE  ← DOCX bảng >6 cột + merged cells → nhiều cell rỗng
     ↓
_table_to_structured_list()
     ↓
Sub-header heuristic: 70%+ bold/empty = sub-header
Với DOCX merged cells: continuation rows xuất hiện với nhiều empty cells
→ actual_data_start quá cao → data_rows[actual_data_start:] = EMPTY
     ↓
structured = ""  (empty string)
     ↓
if structured:  ← FALSE
    result.append(structured)
else:
    # Fallback: remove empty columns, rebuild
    empty_cols = _count_empty_columns(rows)  ← merged cell rows = almost all empty
    cleaned_rows = _remove_empty_columns(rows, empty_cols)  ← XÓA HẾT CỘT
    result.append(_rebuild_table(cleaned_rows))  ← returns "" vì max_cols = 0
     ↓
TABLE CONTENT = MẤT HOÀN TOÀN ❌
```

**3 điểm bug cụ thể:**

| # | File | Vị trí | Bug |
|---|------|--------|-----|
| 1 | `markdown_cleaner.py` | `_table_to_structured_list()` line 153-160 | Sub-header heuristic 70% quá aggressive → xếp data rows thành sub-headers → `actual_data_start` quá cao → không có data rows nào để output |
| 2 | `markdown_cleaner.py` | `_clean_tables()` line 335-342 | Khi `structured = ""`, fallback xóa empty columns → với DOCX merged cells, hầu hết cột bị đánh dấu "empty" → xóa hết → `_rebuild_table` trả về `""` → content mất |
| 3 | `markdown_cleaner.py` | `_count_empty_columns()` line 100-108 | Chỉ check data rows sau header → với DOCX, merged cell continuation rows xuất hiện trong "data" section với hầu hết cells rỗng → cột bị đánh dấu empty sai |

**Vấn đề thứ hai — Thiếu DOCX format config:**

```python
# docling_service.py — chỉ config PDF, DOCX dùng default:
converter = DocumentConverter(
    format_options={
        InputFormat.PDF: PdfFormatOption(pipeline_options=pipeline_options)
        # ← KHÔNG có InputFormat.DOCX config!
    }
)
```

Docling DOCX pipeline dùng direct XML parsing (không cần ML như PDF). Nhưng nếu Docling có `WordFormatOption`, cần kiểm tra và thêm vào.

#### Giải pháp đề xuất

**Fix 1 — An toàn nhất: Thêm "preserve original" fallback trong `_clean_tables()`**
```python
if structured:
    result.append(structured)
else:
    # BUG FIX: Khi structured list conversion thất bại,
    # giữ nguyên bảng gốc thay vì drop silently
    result.extend(table_lines)  # ← keep original, không xóa
```

**Fix 2 — Sửa sub-header heuristic trong `_table_to_structured_list()`**
```python
# Trước: 70% bold/empty = sub-header
if bold_or_empty >= len(row) * 0.7:
# Sau: chỉ rows mà TẤT CẢ non-empty cells đều bold mới là sub-header
non_empty_cells = [c for c in row if c.strip()]
if non_empty_cells and all('**' in c for c in non_empty_cells):
```

**Fix 3 — Thêm debug logging để kiểm tra raw Docling output trước khi clean**
```python
def convert_document_to_markdown(file_path: str) -> str:
    result = converter.convert(file_path)
    raw_markdown = result.document.export_to_markdown()
    logger.debug(f"Raw markdown (first 500 chars): {raw_markdown[:500]}")  # ← thêm
    cleaned_markdown = clean_markdown(raw_markdown)
    return cleaned_markdown
```

**Fix 4 — Kiểm tra nếu Docling có WordFormatOption cho DOCX**
- Cần test: `from docling.document_converter import WordFormatOption` có tồn tại không
- Nếu có: thêm vào `format_options` dictionary

## Decisions

1. **Ưu tiên cải thiện Docling config TRƯỚC** — vì đây là vấn đề chất lượng cốt lõi, ảnh hưởng trực tiếp đến giá trị sản phẩm
2. **UX Progress — dùng giải pháp Hybrid:** Upload progress thật (axios) + Simulated stages + SSE cho future
3. **Thêm Post-processing layer** — tạo module `markdown_cleaner.py` để clean output trước khi trả về frontend
4. **Fix type mismatch** — đồng bộ `ProcessingStage` type với UI steps
5. **[MỚI] Fix CRITICAL bug trong `markdown_cleaner.py`** — silent content drop khi complex table → structured list conversion thất bại. Ưu tiên cao hơn mọi enhancement vì gây mất **toàn bộ** nội dung bảng trong DOCX.
6. **[MỚI] Nguyên tắc "never drop content silently"** — mọi fallback trong markdown_cleaner phải giữ nguyên content gốc, không được trả về empty string.
